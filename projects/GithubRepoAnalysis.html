Github Repository Survival Analysis¶

Johann Miller, Zack Khan, Eashaan Kumar

Introduction¶

Open source projects power the innovations of programmers on a worldwide scale, with companies big and small integrating them into the backbone of their products. However, there is a certain risk with using open source projects, as projects can stop being maintained which leads to bugs being left unfixed and security vulnerabilities unpatched. Also, if you are a programmer who wants to contribute to open source projects, you most likely want to work on a project that will be maintained for a long period of time to ensure your contributions are used by others.

To help potential users of open source projects determine whether they should use/contribute to a specific project, we will be investigating what factors affect the longevity of a github project. Knowing these factors can also help contributors know what to focus on to extend their project’s longevity. Companies like Github that want programmers to continue contributing to open source projects could also use this information as a basis to decide which areas to focus on to increase project longevity as well.

Our initial hypothesis is that the number of unique contributors and number of stars positively correlate with the longevity of a project. We define an abandoned project as a project that has received less than three commits over the last year of our data (2016).

Data Collection¶

We collected data from the following public sources:

  • Github Archive (on Google BigQuery)
  • Github API

In order to analyze repositories, we needed several things: features for each repository (number of commits, number of unique contributors), and a label for each repository (the length of time for which it was maintained). In order to collect this information, we compiled data from Github Archive and Github API on public, open-source repositories that have been abandoned. We start this process by getting repositories from Google BigQuery.

Get repository events data from BigQuery¶

Warning: If you process more than 1 TB in one month on BigQuery, you will be charged. These queries total to about 50 GB.

Github Archive on BigQuery stores event data (pushes, stars, forks, etc.) for every repository from 2011 to the present. In order to do survival analysis, we will select repositories that were created in 2011 and were abandoned in or before 2016. We consider a repository to be abandoned if it receives less than 3 pushes in a year. We also filter out repos that received less than 10 stars in 2011, since we want to analyze somewhat prevalent repositories.

-- get all repos starting in 2011
SELECT repo.id, repo.name, repo.url
FROM [githubarchive:year.2011]
WHERE type = 'CreateEvent'
AND JSON_EXTRACT(payload, '$.ref_type') = '"repository"'
-- save table as final.repos_created_in_2011

-- from the previos result, select all repos that received 10 or more stars in 2011
SELECT *
FROM (
  SELECT archive.repo.id AS repo_id, COUNT(*) AS star_count
  FROM [githubarchive:year.2011] AS archive
  INNER JOIN [final.repos_created_in_2011] AS repos
  ON archive.repo.id = repos.repo_id
  WHERE archive.type = "WatchEvent" -- star events
  GROUP BY repo_id
)
WHERE star_count >= 10
-- save table as final.repos_created_in_2011_10_stars

-- select all of the 2011 repos that had 3 or more pushes in 2016
SELECT *
FROM (
  SELECT archive.repo.id AS repo_id, COUNT(*) AS push_count
  FROM [githubarchive:year.2016] AS archive
  INNER JOIN [final.repos_created_in_2011_10_stars] AS repos
  ON archive.repo.id = repos.repo_id
  WHERE archive.type = "PushEvent"
  GROUP BY repo_id
)
WHERE push_count >= 3
-- save table as final.repos_alive_in_2016

-- select all of the 2011 repos that weren't active in 2016
SELECT *
FROM [final.repos_created_in_2011_10_stars]
WHERE repo_id NOT IN (
  SELECT repo_id
  FROM [final.repos_alive_in_2016]

Collect label and features for each repository¶

Now that we have a table of all the events, we can calculate the labels and features. For each repository we will:

  • Find the creation date and the date the project was abandoned.
  • Choose an arbitrary split between the two dates. The events occuring before the split will be fed to the model, and everything occuring after it will be ignored. The label will be the number of days from the split until the repository is abandoned.
  • We then calculate the features. For each type of event, we calculate:
    • the total number of that event before the split
    • the number of that event in the month before the split
    • the total number of unique users of that event before the split
    • the number of unique users of that event in the month before the split

Splitting the data is an important step. We want to build a model

In [3]:
event_types = set(events.type) # all types of event
activity_event_types = event_types - set(['WatchEvent', 'ForkEvent', 'IssuesEvent']) # maintainence events

# get the label and features for a given repo
def get_repo_data(repo_id):
    
    # get events for this repo
    e = events[events.repo_id == repo_id]
    e = e.sort_values('created_at')

    # get events that show repo is being maintained
    activity_events = e[e.type.isin(activity_event_types)]

    if len(activity_events) == 0:
        logging.debug('skipping, no events: ' + str(repo_id))
        return None
    
    # get first and last activity events
    start_date = activity_events.iloc[0].created_at.date()
    end_date = activity_events.iloc[-1].created_at.date() + datetime.timedelta(days=1)
    total_days_alive = (end_date - start_date).days

    if total_days_alive < 2:
        logging.debug('skipping, < 2 days alive: ' + str(repo_id))
        return None
    
    # select cutoff at random day between start and end
    cutoff_date = start_date + datetime.timedelta(days=random.randrange(1, total_days_alive))
    last_month = cutoff_date - datetime.timedelta(days=30)

    repo_data = {}
    repo_data['repo_id'] = repo_id

    # add label
    repo_data['days_left'] = (end_date - cutoff_date).days

    # add features
    repo_data['days_alive'] = (cutoff_date - start_date).days

    e_prior = e[e.created_at < cutoff_date] # select events before cuttoff
    for event_type in event_types:
        g = e_prior[e_prior.type == event_type] # select events of given type
        repo_data[event_type + '_total'] = len(g)
        repo_data[event_type + '_unique_actors'] = len(g.actor_login.unique())

        g = g[g.created_at > last_month] # select events only in last month
        repo_data[event_type + '_last_month'] = len(g)
        repo_data[event_type + '_last_month_unique_actors'] = len(g.actor_login.unique())
        
    return repo_data

# all of the features to be collected
feature_cols = ['days_alive'] + [
    x + y for x in event_types for y in ['_total', '_unique_actors', '_last_month', '_last_month_unique_actors']
]
data = pd.DataFrame(columns=(['repo_id', 'days_left'] + feature_cols))

# for each repo, collect data
for repo_id in events.repo_id.unique():
    d = get_repo_data(repo_id)
    
    if d is not None:
        data.loc[len(data)] = d # add row to data
        
repo_ids = data.repo_id.unique()

SVM¶

Although the PCA analysis isn't promising, we can try to fit a support vector machine to the data. For classification, SVM selects hyperplanes that minimize error by maximizing the margin between classes of data. For regression, it works similarilty: a hyper plane is fit along the data. If the data is not linear, a kernel function is used to transform the data. Our lifespan label is continuous, so we will use regression.

SVM regression has two major hyper parameters: C , the penalty for error, and epsilon, the margin of tolerance for errors. In order to choose values for the hyper parameters, we can test out a range of options. Here, we use the scikit-learn SVM regression implementation with a polynomial and radial-basis kernel, range of C from 1 to 100, and a range of epsilon from 0.0001 to 0.01. The tests are done with 3-fold cross validation.

In [13]:
scaled_features = preprocessing.normalize(features) # scale features
parameters ={ # set range for hyperparameters
    'C': np.power(10.0,range(-1,3)), 'epsilon': np.power(10.0,range(-4,-1)), 'kernel': ('poly', 'rbf')
}

svr = svm.SVR()
rgr = model_selection.GridSearchCV(svr, parameters, cv=3) # use gridsearch to select hyperparams
scores = model_selection.cross_val_score(rgr, scaled_features, label, cv=3, n_jobs=-1) # score using cross validation
print("r^2: %0.2f (+/- %0.2f)" % (scores.mean(), scores.std() * 2))

This model did not fit the data. The r^2 value tells us that the variance in life span of the repositories is not at all explained by these features. Since these features are not successful, we can consider other aspects of the data.

Analysis of Repository Licenses¶

All repositories have their own licenses. We hypothesize that different licenses might affect user behaviour is different ways, and hence affect the lifespan of the repository. Popular licenses, such as the MIT license, provide more freedom for the users and hence may extend the lifespan of the repository. To get a better understanding, we can perform an ANOVA Test to compare the distribution of each license type. Here is how an ANOVA test is conducted:

Let $\mu_1$ = [true mean lifespan of license type 1]

Let $\mu_2$ = [true mean lifespan of license type 2]

...

Let $\mu_n$ = [true mean lifespan of license type n]

Null Hypothesis: All means are equal

Alternative Hypothesis: Atleast two means are different

$\alpha$ = 0.05

Conclusion¶

We set out to examine how various features affect the lifespan of repositories. After performing various analysis on our data, it is obvious that our data does not predict lifespans on repositories. We are very surprised with these results; we expected that features like the number of commits and unique number of contributors would be valuable indicators for how long a repository will be maintained. Even programming language and license type, which we thought would certainly have some relation with the success of a project, showed to have no significant impact on the lifespan of repositories when put through anova testing.

Since repository survival is an instance of human behavior, we did not expect to be able to model it perfectly. There are many aspects on the human side of this problem that we are blind to with this model. Other factors determining repository survival, like the popularity of certain tech, the friendliness of the collaboration environment, and others could potentially be added as features to the model and lead to improvement. Still, it comes as a surprise that the basic events that make up all of the activity in a repository do not provide valuable information.

It is worth considering that the poor quality of our model may be due to a fault in the way we set up this expiriment. We have identified some potential sources of error: due to the range of the data, we only were able to consider activity between 2011 and 2016. Repositories still being maintained at the end date of our data were removed from the data set. This meant we missed many of the most popular projects, since we could not label them with a lifespan. Secondly, many of the repositories we did consider were not prominent projects. Our requirement of 10 stars in one year was met by many repositories that weren't pushed to beyond the first few days. As can be seen in the PCA plot, these short-lifespan repositories have a seemingly random alotment of features and led to confusion for the model.

We also considered two elements that we thought would certainly impact the maintenence of repositories: programming language and software license. However, ANOVA tests showed both to have abysmal significance for any difference between the populations. It is possible that the aforementioned data collection lead to failed results in these experiments as well.

All in all, these results are very surprising and as such, deserve to be tested more thouroughly being accepted. As future work, we would suggest that any interested party try different sets of features and test different parameters for data collection.
